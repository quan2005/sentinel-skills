# API/Interface Constitution Patterns

API/接口相关宪法的条款模板和验证方式。

当宪法涉及 REST API、库/SDK、CLI 工具、任何模块接口设计时使用。

---

## 核心原则

```
一致性是王。用户讨厌惊讶。

一致性检查清单：
- [ ] 相似操作有相似命名
- [ ] 相似输入有相似格式
- [ ] 相似输出有相似结构
- [ ] 错误格式统一
- [ ] 命名约定统一（camelCase 或 snake_case，不混用）
```

---

## 条款模板

### REST API 结构条款

```
## 条款：RESTful 结构

API 遵循 REST 最佳实践，URL 结构清晰一致。

**验证方式**：
- [ ] 资源使用复数名词：/api/v1/users
- [ ] 嵌套资源：/api/v1/users/{id}/orders
- [ ] HTTP 方法语义正确：
  - GET 读取（幂等）
  - POST 创建
  - PUT 替换（幂等）
  - PATCH 部分更新
  - DELETE 删除（幂等）
- [ ] 查询参数用于过滤，不用于动作
```

### 响应格式条款

```
## 条款：响应格式统一

API 响应格式在整个接口中保持一致。

**验证方式**：
- [ ] 成功响应统一结构：{ data, meta }
- [ ] 错误响应统一结构：{ error: { code, message, details } }
- [ ] 不混用格式（有时 { data } 有时直接返回对象）
- [ ] 状态码符合语义
```

### 状态码条款

```
## 条款：HTTP 状态码

API 使用正确的 HTTP 状态码。

**验证方式**：
- [ ] 200 OK - 通用成功
- [ ] 201 Created - 资源创建成功
- [ ] 204 No Content - 成功无返回（DELETE）
- [ ] 400 Bad Request - 无效输入
- [ ] 401 Unauthorized - 未认证
- [ ] 403 Forbidden - 已认证但无权限
- [ ] 404 Not Found - 资源不存在
- [ ] 409 Conflict - 冲突（如重复）
- [ ] 422 Unprocessable - 语义错误
```

### 分页条款

```
## 条款：列表分页

所有可能返回 >50 项的列表端点支持分页。

**验证方式**：
- [ ] 支持 page/limit 或 cursor/limit 参数
- [ ] 响应包含分页元数据：total, page/nextCursor, hasMore
- [ ] 大数据集使用游标分页
```

### 文档条款

```
## 条款：接口文档

每个公开接口都有完整文档。

**验证方式**：
- [ ] 功能描述（一行）
- [ ] 参数类型说明
- [ ] 返回值说明
- [ ] 使用示例
- [ ] 错误情况说明
```

### 版本管理条款

```
## 条款：API 版本策略

API 使用清晰的版本策略。

**验证方式**：
- [ ] URL 版本化：/api/v1/, /api/v2/
- [ ] 破坏性变更递增主版本号
- [ ] 弃用端点有迁移指南
- [ ] 至少维护一个旧版本
```

### 库/SDK 设计条款

```
## 条款：易用性

库/SDK 可在 5 分钟内上手使用。

**验证方式**：
- [ ] 安装简单（一个命令）
- [ ] 示例代码可复制粘贴运行
- [ ] 默认配置适用于常见场景
- [ ] 高级配置可选而非必需
- [ ] API 表面小而明显
```

### CLI 工具条款

```
## 条款：CLI 标准

CLI 工具遵循命令行标准。

**验证方式**：
- [ ] 支持 --help 和 --version
- [ ] 标准选项：--verbose, --quiet, --config, --output
- [ ] 输出有进度反馈
- [ ] 退出码符合约定（0=成功）
```

## 库/SDK 条款示例

### 简洁 API 条款
```
## 条款：API 简洁

库的公共 API 小而直观，隐藏复杂性。

**验证方式**：
- [ ] 主要操作 ≤ 5 个公开方法
- [ ] 初始化简单（只需必要参数）
- [ ] 配置可选而非必需
- [ ] 内部细节不暴露
```

### 错误处理条款
```
## 条款：清晰错误

库提供清晰、可操作的错误信息。

**验证方式**：
- [ ] 错误类型明确（不抛出通用 Error）
- [ ] 错误消息说明问题原因
- [ ] 错误包含解决建议（如适用）
- [ ] 错误对象包含调试信息
```

## API 检查清单

```
发布前验证：
- [ ] URL 结构 RESTful
- [ ] HTTP 方法语义正确
- [ ] 输入验证完整
- [ ] 认证检查存在
- [ ] 授权检查存在
- [ ] 状态码正确
- [ ] 响应格式一致
- [ ] 错误格式一致
- [ ] 文档完整
- [ ] 示例代码可运行
```

---

## API 条款 Red Flags

```
❌ "API 很好用" — 主观
❌ "REST 风格" — 太宽泛
❌ "返回数据" — 未说明格式

✅ "GET /api/v1/users 返回 { data: User[], meta: Pagination }"
✅ "错误响应格式：{ error: { code, message, details[] } }"
✅ "所有需要认证的端点返回 401 当未提供 token"
```
