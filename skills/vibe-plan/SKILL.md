# Vibe Plan

## 概述

将模糊想法转化为宪法式的执行契约。通过头脑风暴明确需求、定义清晰目标和可验证的成功条款，最终生成宪法文档和 ralph-wiggum 持久迭代执行 prompt。

**核心原则**：宪法即契约 — 定义"做成什么样"的原则性条款，而非具体执行步骤。

## 适用场景

```
┌─────────────────────────────────────────────────────────────┐
│                      你有一个想法                            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │   使用此技能     │
                    └─────────────────┘
```

**适用于**：
- 开发任务（新功能、Bug 修复、重构、测试）
- 内容创作（文章、视频、演讲稿）
- 方案设计（产品方案、技术方案、营销方案）
- 学习计划（技能学习、阅读计划、研究课题）
- 项目规划（活动策划、旅行计划、装修方案）
- 任何需要"把想法变成行动"的场景

**不适用于**：
- 已有详细规范 → 直接执行
- 简单一句话任务 → 直接完成
- 纯探索研究 → 使用 Explore agent

## 信任方程

宪法契约的本质是**信任** — 用户信任这个宪法能准确传达他们的意图，执行 AI 能理解并达成目标。

信任通过**可预测性**和**透明度**建立：
- 用户不应该对宪法内容感到意外
- 用户应该始终知道正在发生什么
- 用户应该感觉到掌控感，而非被拖着走

**每次交互要么建立信任，要么侵蚀信任。没有中间状态。**

### 建立信任的行为

| 行为 | 为什么建立信任 |
|------|---------------|
| 逐段呈现宪法，每段确认 | 用户保持掌控感 |
| 说明拆分/决策的理由 | 透明度增加可预测性 |
| 暴露不确定性而非假设 | 诚实减少惊讶 |
| 请求输入而非默默决定 | 用户参与决策过程 |

### 破坏信任的行为

| 行为 | 为什么破坏信任 |
|------|---------------|
| 一次性输出全部内容 | 用户失去掌控，感到被拖着走 |
| 静默做出范围决策 | 惊讶 = 信任侵蚀 |
| 接受模糊的需求 | 结果不符合预期 |
| 说"应该没问题" | 通常有问题，信任受损 |

### 信任修复

当意识到可能破坏了信任：

```
"抱歉，我应该先和你确认。

关于 [具体点]，我默认做了 [假设]。这可能导致：
- [后果1]
- [后果2]

让我重新问：[问题]"
```

## 四条不可违背的法则

违背这些法则 = 违背宪法契约的本质

### 法则 1：理解先于定义

在写入任何宪法条款前，必须能回答：
- **WHO** — 谁是目标受众/使用者？
- **WHAT** — 要解决什么问题？
- **WHY** — 核心价值是什么？
- **HOW** — 如何验证成功？

```
提问成本 = 2 分钟
错误假设成本 = 2 小时返工

When in doubt, ask. 用户尊重问题，讨厌惊讶。
```

### 法则 2：暴露所有决策

任何关于范围、优先级、拆分的决策都必须说明理由。

**判断标准**：如果你不能 100% 确定它显然正确，就说明它。

**Good**:
```
"我建议将此目标拆分为 3 个独立宪法，因为：
1. 它们可以并行执行
2. 每个部分都有明确的成功标准
3. 拆分后更容易追踪进度

这样拆分可以接受吗？"
```

**Bad**:
```
[默默拆分，直接输出多个宪法文档]
```

### 法则 3：原子化验证

每个宪法条款必须：
1. 描述抽象原则（不规定具体执行步骤）
2. 提供可操作的验证指导
3. 明确"做成什么样"而非"怎么做"

**条款级别**：
- ✅ "价格表在所有设备尺寸上保持良好的可读性"
- ❌ "使用 Tailwind 的 responsive classes 实现响应式"

### 法则 4：契约精神

宪法文档的质量标准不是"能用就行"，而是**"以此为荣"**。

**质量测试**：自问 — **"如果将这个宪法交给另一位 AI 执行，它能理解并达成目标吗？"**

如果答案是"也许"或"大概"，那宪法还不够清晰。

## 核心哲学（补充）

| 原则 | 含义 |
|------|------|
| **一事一宪法** | 每个宪法文档只描述一个原子目标 |
| **宪法即契约** | 定义"成功是什么样"，不规定"怎么做" |
| **可验证性** | 每个条款都有明确的验证指导 |
| **死磕到底** | 配合 ralph-wiggum 持久迭代直到达成 |

## 工作流程

```
你的想法
    │
    ▼
阶段 1: 头脑风暴澄清 (使用 AskUserQuestion)
    │
    ▼
阶段 2: 规模评估拆分 (大型目标需要拆分)
    │
    ▼
阶段 3: 宪法定义 (目标 + 成功条款)
    │
    ▼
阶段 4: 两段式输出 (文档 + 执行 Prompt)
```

### 阶段 1：头脑风暴澄清

使用 `AskUserQuestion` 工具，根据场景选择对应的问题模板。可一次问多个相关问题。

#### 通用 Discovery 问题

```
"在开始之前，帮我理解：

**The Problem（问题）**
1. 当前情况有什么痛点？
2. 是什么触发了这个需求？

**The User（用户）**
3. 谁是具体的使用者/受众？（不是泛泛的"用户"）
4. 他们需要完成什么？

**The Success（成功）**
5. 如果这件事完美达成，会有什么不同？
6. 我们如何知道它成功了？（可衡量）

**The Constraints（约束）**
7. 必须与现有系统集成？
8. 技术栈/格式/风格要求？
9. 时间或资源约束？

让我们从问题（1, 2）和成功（5, 6）开始 — 它们揭示核心。"
```

#### 场景化问题模板

**开发任务**：
1. 这个功能在系统的哪个位置？
2. 需要遵循什么现有模式/约定？
3. 有依赖的其他模块吗？

**内容创作**：
1. 目标读者的现有水平？（初级/中级/高级）
2. 内容重点是讲解/实战/观点？
3. 预期篇幅和深度？

**学习计划**：
1. 已有哪些相关基础？
2. 学习目标：能看懂/能写/深入理解？
3. 预期投入时间？

**方案设计**：
1. 目标规模有多大？（人数/时长/预算）
2. 有什么硬性约束？
3. 成功的最小标准是什么？

#### Discovery 输出格式

完成澄清后，输出理解摘要供用户确认：

```
## 理解摘要

**目标**：[一句话]
**受众**：[具体用户]
**解决**：[核心问题]
**成功**：[可衡量结果]

**范围内**：[要做什么]
**范围外**：[不做什么]

这个理解准确吗？有遗漏吗？
```

**多选题优先，开放题其次**

### 阶段 2：规模评估拆分

**原子目标**（无需拆分）：
- 单个组件/单篇文章/单个活动
- 预估 ≤ 2 小时

**大型目标**（必须拆分）：
- 多模块关联/多阶段项目
- 预估 > 2 小时

**拆分示例**：
```
❌ 写一个技术博客系列
   ↓ 拆分为：
   ✅ distributed_systems_overview_constitution.md      # 系列总览
   ✅ cap_theorem_constitution.md                       # 第一篇：CAP 定理
   ✅ consistency_models_constitution.md                # 第二篇：一致性模型
```

### 阶段 3：宪法定义

逐段呈现目标和宪法条款，每段确认后再继续。

### 阶段 4：两段式输出

1. **宪法文档** → 写入 `.context/`
2. **执行 Prompt** → 调用 `/ralph-wiggum:ralph-loop`

## 宪法文档格式

```markdown
# 目标

<清晰的目标陈述，一句话概括要达成什么>

# 宪法条款

## 条款 1：<标题>

<条款描述 - 抽象原则级别的陈述>

**验证方式**：<如何验证此条款达成>

---

## 条款 2：<标题>

...

# 不做的事项（可选）

<明确不做的范围，防止蔓延>
```

## 文件命名与输出

### 文件路径

`.context/{日期}_{关键标题}_constitution_{序号}.md`

| 字段 | 格式 | 示例 |
|------|------|------|
| 日期 | YYYYMMDD | `20260123` |
| 关键标题 | 目标的核心关键词（英文/拼音） | `price_table` / `wensanzhang` |
| 序号 | 两位数字 | `01`, `02`, `03` |

### 命名示例

| 目标 | 文件名 |
|------|--------|
| 首页价格表 | `20260123_price_table_constitution_01.md` |
| Web Components 文章 | `20260123_web_components_article_constitution_01.md` |
| 学习 Rust | `20260123_rust_learning_constitution_01.md` |
| 团建活动策划 | `20260123_team_building_constitution_01.md` |

### 执行 Prompt 格式

**单个宪法文档**：
```
/ralph-wiggum:ralph-loop "根据 .context/20260123_price_table_constitution_01.md 执行。

完成标准：逐项验证宪法条款，全部达成后输出 '宪法验证通过'。" --completion-promise "宪法验证通过" --max-iterations 10
```

**多个宪法文档**：
```
/ralph-wiggum:ralph-loop "依次执行：
1. .context/20260123_distributed_systems_overview_constitution_01.md
2. .context/20260123_cap_theorem_constitution_02.md
3. .context/20260123_consistency_models_constitution_03.md

完成标准：按顺序执行每个宪法文档，全部条款达成后输出 '全部宪法验证通过'。" --completion-promise "全部宪法验证通过" --max-iterations 20
```

## 宪法文档质量标准

在认为宪法文档"完成"之前，必须通过以下检查。

### 质量检查清单

- [ ] **目标清晰** — 一句话能概括要达成什么
- [ ] **条款原子性** — 每个条款描述一个独立方面
- [ ] **抽象原则** — 条款不规定具体执行步骤
- [ ] **可验证性** — 每个条款都有明确的验证指导
- [ ] **边界明确** — 有"不做的事项"防止蔓延
- [ ] **可执行性** — 另一位 AI 能理解并达成目标
- [ ] **规模适当** — 预估 ≤ 2 小时可完成

### 质量测试

自问：**"如果将这个宪法交给另一位 AI，它能理解并达成目标吗？"**

如果答案是"也许"或"大概"，那宪法还不够清晰。

### 条款级别判断

**Good（抽象原则）**：
- ✅ "价格表在所有设备尺寸上保持良好的可读性和布局比例"
- ✅ "文章使用流畅的中文技术写作风格，避免翻译腔"
- ✅ "代码中不使用 unwrap()，正确处理错误分支"

**Bad（具体步骤）**：
- ❌ "使用 Tailwind 的 md:grid-cols-3 实现响应式布局"
- ❌ "使用 DeepL 翻译后再人工校对"
- ❌ "用 match 语句处理 Result 类型"

### 验证方式级别判断

**Good（判断指导）**：
- ✅ "检查文章结构，四个部分齐全"
- ✅ "对比现有组件，确认字体、颜色、间距一致"
- ✅ "通读全文，语句通顺自然"

**Bad（具体命令）**：
- ❌ "运行 npm test 确保通过"
- ❌ "用 Chrome 浏览器打开检查"
- ❌ "运行 make build 验证编译"

## 交互协议与沟通标准

### 基本协议

1. **使用 `AskUserQuestion` 工具** — 一次问一个或多个问题
2. **无开场白** — 直接进入头脑风暴
3. **自动识别场景类型** — 内部处理，不暴露给用户
4. **大型目标需说明拆分方案** — 获得用户确认
5. **逐段呈现宪法定义** — 每段确认后再继续

### 沟通标准模板

#### 头脑风暴阶段

```
"在开始之前，需要澄清几个问题：

[使用 AskUserQuestion 工具提问]

从这些问题的答案，我理解到：
- **目标**：[一句话概括]
- **受众**：[具体用户]
- **成功标准**：[可衡量]

这个理解对吗？有遗漏吗？"
```

#### 拆分方案呈现

```
"**规模评估**：预估 [时间]

由于 [原因]，建议拆分为 N 个独立宪法：

| 序号 | 名称 | 内容 |
|------|------|------|
| 1 | [名称] | [一句话描述] |
| 2 | [名称] | [一句话描述] |
| 3 | [名称] | [一句话描述] |

拆分方案可以接受吗？"
```

#### 宪法呈现（逐段）

```
"**目标部分**：
[目标陈述]

确认后我继续定义条款。"
```

#### 条款呈现（逐条）

```
"**条款 N：[标题]**
[条款描述]

**验证方式**：[验证指导]

这条条款准确吗？需要调整吗？"
```

#### 不做事项呈现

```
"**不做的事项**（防止范围蔓延）：
- [排除项 1] — [原因]
- [排除项 2] — [原因]

还有其他需要明确排除的吗？"
```

#### 完成呈现

```
"**宪法定义完成！**

**输出文件**：`.context/xxx.md`
**条款数量**：N 条

**执行 Prompt**：
```
[ralph-wiggum 命令]
```

准备好执行了吗？"
```

#### 暴露决策

当需要做出范围或结构决策时：

```
"关于 [决策点]，我建议：
- **决策**：[选择]
- **理由**：[为什么]
- **考虑过的替代**：[其他选项及未选原因]

这符合你的预期吗？"
```

#### 遇到不确定性

当需求模糊或不完整时：

```
"关于 [具体点]，需求不够明确：

选项 A：[假设1] — 适用场景 [说明]
选项 B：[假设2] — 适用场景 [说明]

我倾向于 [选项] 因为 [理由]，但需要你确认。"
```

## NEVER 列表（行为红线）

这些行为会破坏宪法契约的有效性。避免它们绝对化。

| 永远不要 | 为什么 | 改做 |
|---------|-------|------|
| 理解前就写条款 | 会建错宪法 | 先问 WHO/WHAT/WHY/HOW |
| 条款写成具体步骤 | 限制执行自由度，失去宪法意义 | 写抽象原则 + 验证指导 |
| 验证方式是命令 | 环境变化后验证失效 | 写判断指导而非具体命令 |
| 忘记"不做的事项" | 范围蔓延，永无止境 | 明确排除，防止后期加塞 |
| 接受模糊的需求 | 无法定义成功 | 持续追问直到清晰 |
| 跳过规模评估 | 大型项目混乱失控 | >2 小时必须拆分 |
| 一次输出全部条款 | 用户无法确认，失去掌控 | 逐段呈现，每段确认 |
| 说"应该没问题" | 通常有问题 | 明确标注不确定性 |

### Red Flags（警示信号）

| 思维 | 现实 | 应做 |
|------|------|------|
| "这个需求很简单" | 简单也需要澄清理解 | 按流程走 Discovery |
| "用户没说就不问" | 问问题比返工便宜 | 主动澄清模糊点 |
| "先写个初稿再说" | 初稿会成为事实标准 | 第一次就做对 |
| "条款详细点更好" | 详细 = 执行步骤 | 保持抽象原则级别 |

### 常见错误对照（保留参考）

| 错误做法 | 正确做法 |
|----------|----------|
| 写成具体执行步骤 | 写成抽象原则 + 验证方式 |
| 验证方式是具体命令 | 验证方式是判断指导 |
| 忘记"不做的事项" | 加上"不做的事项"防止蔓延 |
| 执行 prompt 格式错误 | prompt 用引号，参数在最后 |

## 示例

### 示例 1：开发任务 - 首页价格表

**用户输入**：
> "我要在首页加个价格表，用 Tailwind，别加新库。"

**头脑风暴问题**：
- 目标用户是谁？(潜在客户)
- 主要目的是什么？(展示价格、促进转化)
- 有几个定价层级？(三个)

**宪法输出**：
```markdown
# 目标

在首页添加一个展示三个定价层级的价格表。

# 宪法条款

## 条款 1：响应式适配

价格表在所有设备尺寸上保持良好的可读性和布局比例。

**验证方式**：在 375px, 768px, 1024px, 1440px 宽度下检查布局。

## 条款 2：样式一致性

价格表与现有设计系统保持一致的视觉风格。

**验证方式**：对比现有组件，确认字体、颜色、间距一致。

## 条款 3：无技术债务

不引入新的依赖，使用项目已有的 Tailwind 工具类。

**验证方式**：检查 package.json 无新增依赖。

# 不做的事项

- 不包含支付功能集成
- 不包含用户选择后的后续流程
```

**执行 Prompt**：
```
/ralph-wiggum:ralph-loop "根据 .context/20260123_price_table_constitution_01.md 执行。

完成标准：逐项验证宪法条款，全部达成后输出 '宪法验证通过'。" --completion-promise "宪法验证通过" --max-iterations 10
```

### 示例 2：内容创作 - 技术文章

**用户输入**：
> "我想写一篇关于 Web Components 的技术文章。"

**头脑风暴问题**：
- 目标读者是谁？(初级/中级/高级前端开发者)
- 文章重点是什么？(原理讲解/实战教程/最佳实践)
- 预期篇幅？(短文/中篇/深度长文)

**宪法输出**：
```markdown
# 目标

撰写一篇面向中级前端开发者的 Web Components 实战教程文章。
```

| 文件 | 何时使用 | 包含内容 |

### 使用规则

**当宪法涉及特定领域时，必须参考对应的领域知识文件**：
- 领域文件包含该领域的常用条款模板
- 提供验证方式的最佳实践
- 列出该领域的 Red Flags（反面示例）

**示例**：
- 用户说"我要添加一个登录功能" → 参考 `domains/security.md`
- 用户说"我要设计一个首页" → 参考 `domains/ui-frontend.md`
- 用户说"我要写一篇技术文章" → 参考 `domains/content-creation.md`
- 用户说"我要学习 Rust" → 参考 `domains/learning-plan.md`
