# Vibe Plan

## 概述

将模糊想法转化为宪法式的执行契约。通过头脑风暴明确需求、定义清晰目标和可验证的成功条款，最终生成宪法文档和 ralph-wiggum 持久迭代执行 prompt。

**核心原则**：宪法即契约 — 定义"做成什么样"的原则性条款，而非具体执行步骤。

## 适用场景

```
┌─────────────────────────────────────────────────────────────┐
│                      你有一个想法                            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │   使用此技能     │
                    └─────────────────┘
```

**适用于**：
- 开发任务（新功能、Bug 修复、重构、测试）
- 内容创作（文章、视频、演讲稿）
- 方案设计（产品方案、技术方案、营销方案）
- 学习计划（技能学习、阅读计划、研究课题）
- 项目规划（活动策划、旅行计划、装修方案）
- 任何需要"把想法变成行动"的场景

**不适用于**：
- 已有详细规范 → 直接执行
- 简单一句话任务 → 直接完成
- 纯探索研究 → 使用 Explore agent

## 核心哲学

| 原则 | 含义 |
|------|------|
| **一事一宪法** | 每个宪法文档只描述一个原子目标 |
| **宪法即契约** | 定义"成功是什么样"，不规定"怎么做" |
| **可验证性** | 每个条款都有明确的验证指导 |
| **死磕到底** | 配合 ralph-wiggum 持久迭代直到达成 |

## 工作流程

```
你的想法
    │
    ▼
阶段 1: 头脑风暴澄清 (使用 AskUserQuestion)
    │
    ▼
阶段 2: 规模评估拆分 (大型目标需要拆分)
    │
    ▼
阶段 3: 宪法定义 (目标 + 成功条款)
    │
    ▼
阶段 4: 两段式输出 (文档 + 执行 Prompt)
```

### 阶段 1：头脑风暴澄清

使用 `AskUserQuestion` 工具，可一次问多个相关问题。

**重点关注**：
- **WHO** — 谁是目标受众/使用者？
- **WHAT** — 要解决什么问题/达到什么效果？
- **WHY** — 为什么要这样做？核心价值是什么？
- **HOW** — 如何验证成功？

**多选题优先，开放题其次**

### 阶段 2：规模评估拆分

**原子目标**（无需拆分）：
- 单个组件/单篇文章/单个活动
- 预估 ≤ 2 小时

**大型目标**（必须拆分）：
- 多模块关联/多阶段项目
- 预估 > 2 小时

**拆分示例**：
```
❌ 写一个技术博客系列
   ↓ 拆分为：
   ✅ distributed_systems_overview_constitution.md      # 系列总览
   ✅ cap_theorem_constitution.md                       # 第一篇：CAP 定理
   ✅ consistency_models_constitution.md                # 第二篇：一致性模型
```

### 阶段 3：宪法定义

逐段呈现目标和宪法条款，每段确认后再继续。

### 阶段 4：两段式输出

1. **宪法文档** → 写入 `.context/`
2. **执行 Prompt** → 调用 `/ralph-wiggum:ralph-loop`

## 宪法文档格式

```markdown
# 目标

<清晰的目标陈述，一句话概括要达成什么>

# 宪法条款

## 条款 1：<标题>

<条款描述 - 抽象原则级别的陈述>

**验证方式**：<如何验证此条款达成>

---

## 条款 2：<标题>

...

# 不做的事项（可选）

<明确不做的范围，防止蔓延>
```

## 文件命名与输出

### 文件路径

`.context/{日期}_{关键标题}_constitution_{序号}.md`

| 字段 | 格式 | 示例 |
|------|------|------|
| 日期 | YYYYMMDD | `20260123` |
| 关键标题 | 目标的核心关键词（英文/拼音） | `price_table` / `wensanzhang` |
| 序号 | 两位数字 | `01`, `02`, `03` |

### 命名示例

| 目标 | 文件名 |
|------|--------|
| 首页价格表 | `20260123_price_table_constitution_01.md` |
| Web Components 文章 | `20260123_web_components_article_constitution_01.md` |
| 学习 Rust | `20260123_rust_learning_constitution_01.md` |
| 团建活动策划 | `20260123_team_building_constitution_01.md` |

### 执行 Prompt 格式

**单个宪法文档**：
```
/ralph-wiggum:ralph-loop "根据 .context/20260123_price_table_constitution_01.md 执行。

完成标准：逐项验证宪法条款，全部达成后输出 '宪法验证通过'。" --completion-promise "宪法验证通过" --max-iterations 10
```

**多个宪法文档**：
```
/ralph-wiggum:ralph-loop "依次执行：
1. .context/20260123_distributed_systems_overview_constitution_01.md
2. .context/20260123_cap_theorem_constitution_02.md
3. .context/20260123_consistency_models_constitution_03.md

完成标准：按顺序执行每个宪法文档，全部条款达成后输出 '全部宪法验证通过'。" --completion-promise "全部宪法验证通过" --max-iterations 20
```

## 交互协议

1. **使用 `AskUserQuestion` 工具** — 一次问一个或多个问题
2. **无开场白** — 直接进入头脑风暴
3. **自动识别场景类型** — 内部处理，不暴露给用户
4. **大型目标需说明拆分方案** — 获得用户确认
5. **逐段呈现宪法定义** — 每段确认后再继续

## 常见错误

| 错误做法 | 正确做法 |
|----------|----------|
| 写成具体执行步骤 | 写成抽象原则 + 验证方式 |
| 验证方式是具体命令 | 验证方式是判断指导 |
| 忘记"不做的事项" | 加上"不做的事项"防止蔓延 |
| 执行 prompt 格式错误 | prompt 用引号，参数在最后 |

## 示例

### 示例 1：开发任务 - 首页价格表

**用户输入**：
> "我要在首页加个价格表，用 Tailwind，别加新库。"

**头脑风暴问题**：
- 目标用户是谁？(潜在客户)
- 主要目的是什么？(展示价格、促进转化)
- 有几个定价层级？(三个)

**宪法输出**：
```markdown
# 目标

在首页添加一个展示三个定价层级的价格表。

# 宪法条款

## 条款 1：响应式适配

价格表在所有设备尺寸上保持良好的可读性和布局比例。

**验证方式**：在 375px, 768px, 1024px, 1440px 宽度下检查布局。

## 条款 2：样式一致性

价格表与现有设计系统保持一致的视觉风格。

**验证方式**：对比现有组件，确认字体、颜色、间距一致。

## 条款 3：无技术债务

不引入新的依赖，使用项目已有的 Tailwind 工具类。

**验证方式**：检查 package.json 无新增依赖。

# 不做的事项

- 不包含支付功能集成
- 不包含用户选择后的后续流程
```

**执行 Prompt**：
```
/ralph-wiggum:ralph-loop "根据 .context/20260123_price_table_constitution_01.md 执行。

完成标准：逐项验证宪法条款，全部达成后输出 '宪法验证通过'。" --completion-promise "宪法验证通过" --max-iterations 10
```

### 示例 2：内容创作 - 技术文章

**用户输入**：
> "我想写一篇关于 Web Components 的技术文章。"

**头脑风暴问题**：
- 目标读者是谁？(初级/中级/高级前端开发者)
- 文章重点是什么？(原理讲解/实战教程/最佳实践)
- 预期篇幅？(短文/中篇/深度长文)

**宪法输出**：
```markdown
# 目标

撰写一篇面向中级前端开发者的 Web Components 实战教程文章。

# 宪法条款

## 条款 1：内容结构清晰

文章包含背景介绍、核心概念、实战示例、最佳实践四个部分。

**验证方式**：检查文章结构，四个部分齐全。

## 条款 2：代码可运行

文中的所有代码示例都可以直接运行并产生预期效果。

**验证方式**：逐个运行代码示例，验证输出正确。

## 条款 3：中文表达自然

文章使用流畅的中文技术写作风格，避免翻译腔。

**验证方式**：通读全文，语句通顺自然。

# 不做的事项

- 不涉及 Shadow DOM 底层实现原理
- 不讨论各大框架的 Web Components 集成
```

**执行 Prompt**：
```
/ralph-wiggum:ralph-loop "根据 .context/20260123_web_components_article_constitution_01.md 执行。

完成标准：逐项验证宪法条款，全部达成后输出 '宪法验证通过'。" --completion-promise "宪法验证通过" --max-iterations 10
```

### 示例 3：学习计划 - 学习 Rust

**用户输入**：
> "我想学习 Rust 编程语言。"

**头脑风暴问题**：
- 有什么编程基础？(已有哪些语言经验)
- 学习目标是什么？(能看懂代码/能写项目/深入理解)
- 预期投入时间？(一周速成/系统学习)

**宪法输出**：
```markdown
# 目标

在一个月内掌握 Rust 基础语法和核心概念，能够独立完成中等复杂度的命令行工具开发。

# 宪法条款

## 条款 1：掌握所有权系统

理解 Rust 的所有权、借用、生命周期核心概念。

**验证方式**：能够解释所有权规则，并通过 Rustlings 相关练习。

## 条款 2：能够独立开发

使用 Rust 完成一个命令行工具项目。

**验证方式**：项目能够编译通过、正常运行、处理常见错误情况。

## 条款 3：理解错误处理

掌握 Rust 的 Result/Option 错误处理模式。

**验证方式**：代码中不使用 unwrap()，正确处理错误分支。

# 不做的事项

- 不深入学习异步编程(async/await)
- 不涉及宏编程
```

**执行 Prompt**：
```
/ralph-wiggum:ralph-loop "根据 .context/20260123_rust_learning_constitution_01.md 执行。

完成标准：逐项验证宪法条款，全部达成后输出 '宪法验证通过'。" --completion-promise "宪法验证通过" --max-iterations 10
```

### 示例 4：方案设计 - 团建活动策划

**用户输入**：
> "我要策划一次团队团建活动。"

**头脑风暴问题**：
- 团队规模？(人数)
- 活动时长？(半天/全天/多日)
- 预算范围？(人均预算)
- 活动偏好？(户外/室内/运动/放松)

**宪法输出**：
```markdown
# 目标

# 宪法条款

活动设计确保所有成员都能参与，无身体原因的排除项。

**验证方式**：检查活动流程，确认无高强度/危险性项目。

## 条款 2：预算可控

总预算在规定范围内，无隐性费用。

**验证方式**：核对费用明细，总额不超预算。

## 条款 3：有备选方案

关键环节有备选方案（天气、场地等突发情况）。

**验证方式**：检查是否有 Plan B。

# 不做的事项

- 不涉及过夜住宿
- 不安排需要专业培训的活动
```
