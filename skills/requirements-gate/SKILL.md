# 需求门禁 Requirements Gate

把模糊的开发需求变成一份无歧义、用户视角的 **story.md**：只回答"谁、要什么、为什么、做到什么程度算成功"，不碰"怎么实现"。实现方案（技术选型、NFR 取舍、依赖与架构影响）交给下游的 brainstorming 产出 design.md。两层分工：**story.md 是意图契约，design.md 是方案契约**。

## 两层分工

| 文件 | 视角 | 谁产出 | 内容 |
|---|---|---|---|
| **story.md** | 用户视角（"要什么"） | 本技能 + 用户确认 | 背景、目标与假设、验收标准(AC)、非目标、范围 |
| **design.md** | 实现视角（"怎么做"） | brainstorming / 开发阶段 | 技术方案、NFR 取舍、依赖与影响面、架构决策 |

铁律：**story.md 里不出现实现细节**。AC 写成用户可观察的行为，不写技术手段。需要实现知识才能回答的问题（用什么存储、如何回滚、性能上限多少），不在本技能职责内——记入交棒清单，由 design.md 解决。

## 三条核心原则

1. **先补上下文，再评审。** 没有检索过仓库和历史就给的建议，只是听起来正确的通用 PM 话术。每条判断标注 `[证据]`（附文件路径/出处）或 `[推测]`。
2. **不给最终判决。** 不输出"通过/不通过"或"8/10 分"，只输出 readiness 等级、缺口、风险和需要用户拍板的问题。
3. **story.md 是契约不是记录。** 下游验收门禁拿 story.md 的 AC 与非目标核对"做的是不是用户要的"。AC 编号稳定、每条独立可测、非目标明确——否则契约无法机械核对。

## 门禁分级

先分类再决定评审强度：

| 级别 | 适用 | 动作 |
|---|---|---|
| **L0 跳过** | 非开发请求（提问、讨论、运维）；已有 approved story.md 的任务延续；用户标注 `[skip-gate]` / `[跳过门禁]` | 忽略门禁，正常响应 |
| **L1 轻量** | 琐碎修复：typo、改文案、单行 bug、配置调整 | 简版 story.md（一句话需求 + 1–2 条 AC），不提澄清，2 分钟内完成 |
| **L2 标准** | 常规新功能 / 功能变更 | 完整流程 Step 1–6 |
| **L3 深度** | 触碰任意一项：数据契约 / 权限模型 / 计费定价 / 对外 API / 不可逆迁移 / 跨团队指标 | 完整流程 + 多角色评审（见 references/review-dimensions.md）。注意：这些高风险维度本身属实现层，门禁只负责**识别并标记**到交棒清单，方案由 design.md 定 |

拿不准取高不取低。

## 流程

### Step 1 — 上下文收集（检索先于评审）

优先用 `rga` 检索：仓库内相关模块/接口/配置；`stories/` 及 brainstorming 输出目录下的历史 story.md / design.md（重复、冲突、可复用结论）；AGENTS.md / docs 既有约定。证据（文件路径 + 一句话结论）写入 story.md 背景节。检索为空也是信息——全新领域，假设风险更高。

### Step 2 — 模糊性清除

读 `references/ambiguity-checklist.md`，扫描需求原文的模糊表达（"体验好一点""支持更多场景"……）。每处输出四件套：**模糊类型 → 执行风险 → 澄清问题 → 可直接替换进 story.md 的改写**。改写须是用户视角的可观察行为，不引入实现手段。只说"这里不清楚"是失职。

### Step 3 — 缺口扫描与归类

扫描需求是否存在缺口，并按层归类——这是两层分工的关键一步：

| 缺口落点 | 例子 | 处理 |
|---|---|---|
| **意图层**（用户能答、且影响"要什么"） | 目标用户是谁？成功长什么样？哪些明确不做？ | 留在门禁内，进 Step 4 澄清 |
| **实现层**（需实现知识、属"怎么做"） | 用什么存储？如何回滚降级？性能上限？依赖哪些系统？ | **不在门禁内解决**，记入「交棒清单」转 design.md |

实现层缺口（含 L3 的高风险维度：回滚、降级、隐私、可观测性、依赖影响）不要试图在 story.md 里拍板——识别出来、标记清楚、交出去。门禁的价值是**不让实现层缺口被忽略**，而非替用户和 AI 做实现决策。

### Step 4 — 清晰度判定与路由

按意图层缺口的性质路由：

| 缺口性质 | 特征 | 路由 |
|---|---|---|
| **方案级未知** | 目标本身不明、多个可选方向、用户自己也没想清楚 | **交棒 superpowers brainstorming**：带上下文梳理，由它产出 design.md（并可反哺 story.md 的目标澄清） |
| **参数级缺口** | 方向明确，只缺意图层的边界/默认值 | 门禁直接澄清：一轮最多 5 个选项题，能用合理默认值兜底的不问 |

交棒不是简单转手——把 Step 1 检索证据、Step 2 模糊点、Step 3 的**交棒清单**一并带过去。带着证据进 brainstorming 和空手进，产出质量差一个量级。用户跳过的澄清：显式默认值 + 写入 story.md「待确认」，绝不静默假设。

### Step 5 — 产出 story.md

用 `assets/story-template.md`。存放位置跟随 brainstorming 输出约定，皆无约定时默认 `stories/<YYYYMMDD>-<slug>/story.md`，design.md 同目录。初始 `status: clarifying`。

AC 规则：
- 每条格式 **当 {前置条件}，{用户操作}，应 {用户可观察结果}** ——不可观察的不是 AC，是愿望；不含实现手段
- 编号 AC-1、AC-2… 一旦发出**不复用不重排**，废弃的标 `[已废弃]`
- AC + 非目标共同界定范围：AC 给下游判"不漏/不少"，非目标给下游判"不多"

### Step 6 — 门禁判定与交棒

向用户输出（一屏内）：readiness 等级（`草稿`/`待澄清`/`可开发`，定义见 references/review-dimensions.md）；最优先 3–5 个意图层缺口（按风险排序，带 `[证据]`/`[推测]`）；需用户拍板的问题；**交棒清单**（移交 design.md 的实现层问题）。

用户确认后 `status` 改为 `approved`。**approved 之前不写实现代码。** 随后：

- 方案细化与开发：交 brainstorming 产出 design.md，再接 writing-plans（或 spec-kit `/plan`），把 story.md 与 design.md 路径写进计划首行
- 开发完成后：commit 前由 **verification-gate** spawn 独立 subAgent，以 story.md 的 AC 与非目标核对"做的是不是用户要的"、design.md 的范围核对"有没有做多/偏"，产出 verify-report.md；通过后翻 `verified`

## Rationalizations — 想抄近道时读这里

| 借口 | 纠正 |
|---|---|
| "这需求很简单，不用走门禁" | 简单 → L1，2 分钟留档，不是跳过。"简单"本身是未验证的假设 |
| "顺手把实现方案也写进 story" | story 只写"要什么"。实现细节进交棒清单 → design.md，混进来就是越层 |
| "用户很急" | 越急越容易返工。急 → 少问、默认值给得果断，而不是不问 |
| "我理解用户的意思了" | 理解 ≠ 共识。写下来被用户确认过的才是共识 |
| "上下文已经够了" | 没有 rga 检索记录就是不够。证据 = 文件路径，不是感觉 |
| "需求不清晰，我先替用户猜个方案" | 方案级未知交棒 brainstorming，门禁不替用户发明需求 |
| "NFR/回滚这些我顺手定了" | 那是实现层决策，标记到交棒清单交给 design.md，别在 story 里拍板 |
| "用户没回澄清，先做了再说" | 显式默认值 + 写入待确认，让假设可见可推翻 |

## Verification — status 改为 approved 前自检

- [ ] story.md 全文无实现细节：AC 是用户可观察行为，未混入技术手段
- [ ] 实现层缺口已归入交棒清单，未在 story 里擅自拍板
- [ ] 每条 AC 独立可测、结果可观察、编号唯一
- [ ] 原文模糊表达已全部改写或列入「待确认」
- [ ] 「非目标」非空
- [ ] 所有判断带 `[证据]`/`[推测]` 标注
- [ ] 「待确认」清零，或用户已显式接受默认值
- [ ] 用户明确说了"确认/通过/可以开发"——沉默不等于确认
