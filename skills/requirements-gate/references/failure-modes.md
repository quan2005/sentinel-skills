# 五种翻车原型 → 五道防线

PM 输入的五种典型翻车。每种给识别信号、为什么危险、防线动作。识别是第一步——信号一出现就启动对应防线。

## A · 解决方案当需求

**信号**：「做一个 X 功能」「加个 Y 按钮」「老板说要做 Z」「我们需要一个 AI 助手」——句式是方案/名词，不是用户问题。

**为什么危险**：方案是某个人对问题的一个猜测。直接做方案 = 把一个人的猜测当成全部真相，跳过了"这是不是真问题、是不是最好的解法"。

**防线**：脊柱 Q1 拒绝方案输入 → 反推 JTBD。不是 block 一次，而是 **block + 重述真实问题 + 展开方案空间**。让 PM 看到老板的方案只是解空间里的一个候选。
> "我注意到『X』是一个方案，不是一个用户问题。先不评价方案好坏，但要做出能落地的需求，必须先确认：这个 X 是为了让谁、在做什么事的过程中、解决什么卡点？"

## B · 用户 = 所有人

**信号**：「用户希望」「很多用户反馈」「大家都觉得」——主语没有定语，泛指全体。

**为什么危险**：为所有人设计 = 为没有人设计。不同角色在不同场景的卡点完全不同，混在一起会得到一个谁都不满意的平均方案。

**防线**：脊柱 Q1 连环追问到具体角色 + 具体场景。一次问不出来就追第二轮：
> "『用户』是哪一类？给一个具体角色（如『中小公司的运营』『独立设计师』），不要说『所有用户』。这类用户最高频做的 3 件事是什么？这个需求是为了让他们在哪一件事上更顺？"

## C · 业务目标硬转译

**信号**：KPI / OKR / 「次留 +5%」「提升 X% 转化」直接当需求接。

**为什么危险**：KPI 是结果，不是需求。从"要 +5% 留存"到"做什么"之间隔着无数条因果路径，直接跳过去等于赌一条没验证的因果链。

**防线**：入口走 BRIDGE（`bridge-step.md`），强制 PM 写 ≥2 个用户行为变化假设，对比解空间，再锁定。让 PM 亲眼看见"选错假设 = 做错产品"。

## D · AI 自由发挥

**信号**：需求里没有「不做的事」。在 AI coding 时代，没有边界 = 授权 AI 自由发挥。

**为什么危险**：AI/工程师会填补一切未明确的空白，且填补方式不可预测。没写"不做对话历史"，就可能凭空多出一个对话历史模块——超范围、增成本、埋 bug。

**防线**：输出闸把 Won't（三类边界）设为 story 模板必填字段。空 = fail，不出。这是给下游 verification-gate 判"不多"的基准。

## E · 看起来全但模糊

**信号**：8 页 PRD，但满是「更方便」「性能良好」「体验流畅」「支持更多场景」——长度有了，可执行性没有。

**为什么危险**：模糊表达让工程、设计、数据各自按不同理解执行，分歧到验收才爆发。篇幅制造了"已经想清楚"的假象，最危险。

**防线**：输出闸 INVEST 6/6 + GWT 硬门。每条验收标准必须可测、带数字阈值。模糊词必须改写。

### 模糊词清除表（原型 E 配套）

扫描 story 正文，以下词命中即改写为可观察、可测的表达：

**中文**：体验好 / 好用 / 流畅 / 快 / 稳定 / 优化 / 提升 / 增强 / 减少 / 支持更多 / 各种场景 / 灵活 / 智能 / 合理 / 适当 / 尽量
**英文**：better / easier / faster / seamless / scalable / robust / flexible / improve / enhance / various / appropriate
**度量类**（出现指标名但无定义/数字即模糊）：体验 / 效率 / 质量 / 稳定性 / 转化率 / 性能

改写方向：谁 + 在什么场景 + 可观察的行为变化 + baseline→目标数字。例：「提升配置体验」→「规则配置完成率 60%→80%」。
