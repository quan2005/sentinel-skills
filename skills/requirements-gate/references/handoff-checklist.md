# 实现层交棒清单 Handoff Checklist

本清单是 Step 3 归类缺口时的参考：逐项判断需求是否触及该维度，触及则记入 story.md 第 7 节「交棒清单」，移交 design.md 解决。**门禁不在这些维度上拍板**——它们需要实现知识，属"怎么做"，由 brainstorming/开发阶段定方案。门禁的职责是确保它们不被忽略。

为什么单列：这些恰是上线后代价最高、又最容易在"用户视角需求"里被漏掉的维度。用户说"加个导出功能"，不会主动说"导出失败怎么降级、大文件性能上限多少、临时文件多久清理"——但这些不定，上线就是事故。门禁把它们从用户需求里**识别出来打包交棒**，是两层分工里最有价值的一环。

## 维度清单

| 维度 | 触发判断（需求是否涉及） | 漏掉的典型后果 |
|---|---|---|
| 性能 / 容量 | 有数据量、并发、响应时间的隐含预期吗 | 演示流畅，真实数据卡死 |
| 安全 / 权限 | 谁能用、有无越权路径 | 普通用户看到管理数据 |
| 数据 / 隐私 | 涉及个人/敏感信息？留存与脱敏 | 合规风险，日志泄露 PII |
| 可靠性 / 降级 | 依赖方挂了怎么办、有无兜底 | 第三方故障拖垮整功能 |
| 可观测性 | 出问题靠什么定位 | 线上故障只能靠用户截图排查 |
| 回滚策略 | 出问题能否回退、数据变更可逆吗 | 带迁移的功能无法回滚 |
| 兼容性 | 影响旧客户端/已有数据/现有调用方吗 | 升级后老数据打不开 |
| 多语言 / 多地区 | 文案、时区、货币、法规差异 | 海外用户时间显示错误 |
| 运营 / 客服 | 增加人工负担、需要 SOP 吗 | 客服不知道新功能怎么答 |
| 风控 / 滥用 | 会被刷吗、批量滥用代价 | 导出接口被爬虫薅崩 |
| 成本 | 算力/存储/三方调用费随量涨吗 | 量上来后账单失控 |

## L3 深度需求的强制交棒项

需求触碰以下任一项（即 L3 判定条件），对应维度**必须**进交棒清单，不允许略过：

- 数据契约变更 → 兼容性、回滚策略
- 权限模型变更 → 安全/权限、可观测性
- 计费/定价 → 成本、风控/滥用、数据/隐私
- 对外 API → 兼容性、性能、安全/权限
- 不可逆迁移 → 回滚策略（迁移失败的处置预案）

## 边界提醒

清单里没有"该选哪个方案"的答案，只有"该问哪些问题"。把维度作为问题交出去，而不是替 design.md 给结论——一旦门禁开始写"建议用 Redis 缓存""回滚用蓝绿部署"，就越层了。
