# 模糊性清除清单 Ambiguity Checklist

模糊表达的危害不是"写得不好"，而是工程、设计、数据各方会按各自理解执行，分歧在验收时才暴露。门禁的职责是把每一处模糊表达转成**可直接替换进 story.md 的具体文本**。

## 高危模糊词表

扫描需求原文，命中即处理：

**中文**：体验好 / 好用 / 易用 / 流畅 / 快 / 稳定 / 优化 / 提升 / 改善 / 增强 / 减少 / 降低 / 支持更多 / 各种场景 / 兼容 / 灵活 / 可扩展 / 智能 / 自动 / 合理 / 适当 / 尽量 / 必要时 / 等等 / 之类

**英文**：better / easier / faster / optimized / seamless / scalable / robust / flexible / improve / enhance / reduce / support / handle / various / appropriate / etc.

**度量类**（出现指标名但无定义即算模糊）：用户体验 / 效率 / 质量 / 稳定性 / 转化率 / 性能

## 五种模糊类型

| 类型 | 特征 | 例子 |
|---|---|---|
| 无主体 | 不知道"谁"受益 | "提升体验" —— 谁的体验？ |
| 无度量 | 变化不可观察 | "更快" —— 多快算快？从哪测到哪？ |
| 无边界 | 范围开放 | "支持更多格式" —— 哪几种？不含哪些？ |
| 隐含假设 | 默认了未声明的前提 | "导出 PDF" —— 默认含图片？含历史数据？ |
| 无失败定义 | 只描述成功路径 | "自动同步" —— 同步失败时呢？ |

## 输出格式（每处模糊表达四件套）

```
原文：「{模糊片段}」
1. 模糊类型：{上表五类之一}
2. 执行风险：{各方会如何按不同理解执行，导致什么分歧}
3. 澄清问题：{一个具体问题，最好带选项}
4. 改写建议：{可直接替换进 story.md 的文本；信息不足时给带显式默认值的版本}
```

## 示例

**示例 1**

原文：「导出体验要好一点」
1. 模糊类型：无主体 + 无度量
2. 执行风险：前端可能理解为加动画，后端可能理解为提速，验收时双方都说做完了
3. 澄清问题：体验好指的是 (a) 导出耗时短 (b) 进度可见可取消 (c) 排版还原度高？可多选
4. 改写建议：「单篇导出在 3 秒内完成；超过 3 秒展示进度条并支持取消（AC 化后为 AC-n）」

**示例 2**

原文：「支持批量操作」
1. 模糊类型：无边界 + 无失败定义
2. 执行风险：批量上限、部分失败语义未定，极端输入会在线上暴露
3. 澄清问题：批量上限多少？部分失败时 (a) 整体回滚 (b) 跳过失败项并报告？
4. 改写建议：「单次批量上限 100 条 [推测：默认值，待确认]；部分失败时跳过失败项，结束后返回成功/失败明细」

## 收尾规则

- 改写后通读一遍 story.md 正文：词表中的词不应再出现在第 3/5 节（背景叙述中出现可接受）
- 用户未答复的澄清问题 → 改写采用显式默认值并标 `[推测]`，同步写入第 9 节「待确认」
