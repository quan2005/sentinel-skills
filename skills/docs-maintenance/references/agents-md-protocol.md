# AGENTS.md 增量修订协议

五步：识别变更 → 增量修订 → 版本定级 → 写入前校验 → 传播检查与同步影响报告。哪怕只改一条措辞，也走完整的定级与校验流程。

## Step 1 — 识别变更

两个来源交叉印证：verified story 的 AC/非目标 与 design.md 的影响面/范围；本次 git diff。关注五类：新模块、新命令、新事件、新约定、废弃项。

契约（story/design）与 diff 对不上（一方有一方无）→ 不写入文档，提示用户回验收环节复核。

## Step 2 — 增量修订

- 新增条目追加到所属章节末尾；变更条目原位修改；废弃条目标注「已被 X 取代（{日期}）」，确认无引用后方可移除
- 禁止全量重写
- **条目写作规范**：每条约定 = 简洁的名称行 + 不可妥协的规则本体 + 理由（仅当理由不自明时写）
- **约束分级措辞**：必须（MUST）/ 应当（SHOULD，须附允许例外的条件）/ 可以（MAY）。清除"尽量、酌情、合理、适当"等不可检验的模糊词——约定必须陈述性、可检验，否则它约束不了任何 Agent
- 关键信息缺失：写 `TODO({字段}): 说明`，登记到同步影响报告的 deferred 项，不编造
- AGENTS.md 不存在时初始化（建议章节：项目概述 / 模块与命令 / 约定与规范 / 事件与流程），版本从 1.0.0 起

## Step 3 — 版本号语义化定级

头部维护 `version` 与 `last-modified`（ISO 日期 YYYY-MM-DD）：

| 级别 | 触发条件 |
|---|---|
| MAJOR | 不兼容变更：约定被移除、反转或重定义 |
| MINOR | 新增章节或条目；对既有指引的实质性扩充 |
| PATCH | 澄清、措辞、错别字等非语义精修 |

定级有歧义时：先写出判断理由，再定级。

## Step 4 — 写入前校验

全部通过才允许落盘：

- [ ] 无残留的空槽位；TODO 均为 `TODO({字段}): 说明` 格式且已登记 deferred
- [ ] 版本行与同步影响报告一致，日期为 ISO 格式
- [ ] 本次新增/修改的条目逐条过检：陈述性、可检验、无模糊词、约束级别明确
- [ ] 格式：标题层级不升降、章节间单空行、无行尾空白

## Step 5 — 一致性传播检查与同步影响报告

rga 检索其余六类文档（ARCH / DESIGN / README / llms.txt / 使用 / 开发说明）中引用本次变更条目的位置：一致 → 记「已检查，无需传播」；不一致 → 一并增量修正（这是「本次变更不涉及的章节不动」的唯一例外，范围严格限于被引用处）；无法可靠自动修正的标 ⚠ 待人工。

顶部维护 HTML 注释形式的同步影响报告，每次更新覆写：

```html
<!-- Sync Impact Report
version: 1.2.0 → 1.3.0 (MINOR)
modified: 模块与命令（新增 docs-maintenance）；「分支约定」改名「Git 约定」
added: 无
removed: 无
propagation: ARCH.md §7 ✅ 已同步；docs/dev-guide.md ⚠ 待人工确认
deferred: TODO(RATIFICATION_DATE): 待用户提供
date: {YYYY-MM-DD}
-->
```
